## Wired release-please for auto-versioning on main push

Replaced the tag-triggered release workflow with a release-please-driven one. Push to main now opens/maintains a "Release PR" that bumps `package.json` + prepends to `CHANGELOG.md`. Merging that PR creates the tag, the GitHub Release, and (in the same workflow run) the build job uploads `savesmanager-output.zip` + deploy scripts as assets.

### Why refactor: and chore: trigger releases
Verified in release-please source (`src/strategies/base.ts`): release is skipped only when `conventionalCommits.length === 0` OR `changelogEmpty(body)` returns true (which is just a `body.split('\n').length <= 1` check). The default versioning strategy (`src/versioning-strategies/default.ts`) bumps major for breaking, minor for `feat`, and **patch for any other commit** that makes it through. So adding `refactor` + `chore` to `changelog-sections` with `hidden: false` is sufficient — they render into the body, body is non-empty, default strategy bumps patch.

### Files
- `release-please-config.json` — `release-type: node`, `package-name: savesmanager`, `bump-minor-pre-major: true`. changelog-sections enables feat/fix/perf/refactor/chore (visible, release-triggering); hides docs/test/build/ci.
- `.release-please-manifest.json` — seeded `{".": "0.2.0"}` to match current package.json.
- `.github/workflows/release.yml` — rewrote. Now triggers on `push: branches: [main]`. Two jobs: `release-please` (runs the action) and `build` (conditional on `release_created == 'true'`, builds + uploads via `gh release upload --clobber`). Top-level `permissions: contents: write, pull-requests: write`.

### Gotcha avoided
GITHUB_TOKEN-created tags don't trigger other workflows, so the tag-listening pattern from the previous release.yml wouldn't have fired off release-please's tag. Solution: keep release-please and the artifact build in the **same workflow run**, gated by the action's `release_created` output.
