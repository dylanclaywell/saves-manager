## Set up Pi-targeted release pipeline for SavesManager

Wired the project for tagged GitHub releases that produce a runnable Nuxt SSR artifact for a Raspberry Pi Zero 2W (Docker was ruled out due to 512MB RAM — running Node directly under systemd instead).

### What landed
- `.github/workflows/release.yml` — on `v*` tag push (or manual `workflow_dispatch`): `npm ci` → `npm run build` → zip `.output/` → attach `savesmanager-output.zip`, `savesmanager.service`, `update.sh`, `install.sh` to the GitHub Release (softprops/action-gh-release@v2). Generates release notes automatically.
- `deploy/savesmanager.service` — systemd unit. Runs `node .output/server/index.mjs` as user `pi`, WorkingDirectory `/opt/savesmanager`, `HOST=0.0.0.0 PORT=3000`, `Restart=on-failure`.
- `deploy/update.sh` — fetches `savesmanager-output.zip` from `releases/latest/download/`, swaps `.output/` atomically (keeps previous as `.output.old`), restarts the service. Self-elevates with sudo. Configurable via `SAVESMANAGER_REPO`, `SAVESMANAGER_DIR`, `SAVESMANAGER_SERVICE` env vars.
- `deploy/install.sh` — first-time bootstrap. Installs Node 20 from NodeSource if missing, creates `/opt/savesmanager`, templates the systemd unit with the resolved node path and user, enables the service, then calls update.sh for the first download. Bootstrap-friendly: if `update.sh`/`savesmanager.service` aren't next to it (e.g. `curl | sudo bash`), it fetches them from the latest release.
- `.gitattributes` — forces LF on `*.sh`, `*.service`, and everything under `deploy/` and `.github/` so Windows-edited scripts run on the Pi.
- `.gitignore` — removed `.memories` so goldfish memory commits to source control.

### Why this shape
- Nuxt 3 SSR isn't static — `.output/` is a Node bundle that needs `node` at runtime, but Nitro bundles all runtime deps into `.output/server/node_modules/`, so the zip is self-contained (no `npm install` on the Pi).
- Current deps (nuxt, vue, vue-router, env-paths) are pure JS, so building on `ubuntu-latest` x64 and running on ARM Pi works. If a native-binding dep is added later (e.g. better-sqlite3), the workflow will need an ARM matrix.

### Pi deploy flow (end state)
1. First time: `curl -fsSL https://github.com/dylanclaywell/saves-manager/releases/latest/download/install.sh | sudo bash`
2. Updates: `sudo /opt/savesmanager/update.sh`
3. Service mgmt: `systemctl status savesmanager`, `journalctl -u savesmanager -f`
